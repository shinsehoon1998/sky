# PRD (Product Requirements Document)

## 고객정보 등록 및 동의서 출력 업무 자동화 시스템

---

### 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| **프로젝트명** | 고객정보 등록 및 동의서 출력 업무 자동화 시스템 |
| **문서 버전** | v2.0 |
| **작성일** | 2026-06-02 |
| **목표 플랫폼** | 웹 애플리케이션 (Cloudflare Pages + Hono) |

---

### 2. 문제 정의 (Problem Statement)

**현재 업무 방식의 문제점:**
- 엑셀 파일에 저장된 고객 정보(주민등록번호, 이름, 연락처)를 수동으로 영업용 전산 시스템에 한 건씩 입력
- 각 고객마다 동의서를 개별 출력해야 하는 번거로움
- 240건 이상의 데이터(50~59년생 고객)를 수작업으로 처리할 경우 많은 시간과 인적 오류 발생

**해결하고자 하는 목표:**
엑셀 파일을 업로드하면 고객 데이터를 자동으로 파싱하여 웹 브라우저 기반 전산 시스템에 등록하고, 고객별 동의서를 출력하는 업무 자동화 프로그램 개발

#### 7.2 DB 스키마 (Cloudflare D1)

```sql
-- 처리 이력 테이블
CREATE TABLE processing_history (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  batch_id TEXT NOT NULL,
  resident_number_encrypted TEXT NOT NULL,  -- 암호화된 주민번호
  name TEXT NOT NULL,
  phone TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending',   -- pending, processing, completed, failed
  consent_printed INTEGER DEFAULT 0,        -- 동의서 출력 여부
  error_message TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 배치 처리 테이블
CREATE TABLE processing_batch (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  batch_id TEXT UNIQUE NOT NULL,
  total_count INTEGER NOT NULL,
  completed_count INTEGER DEFAULT 0,
  failed_count INTEGER DEFAULT 0,
  status TEXT NOT NULL DEFAULT 'pending',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_batch_id ON processing_history(batch_id);
CREATE INDEX idx_status ON processing_history(status);
```

### 11. 중요 고려사항 (주의사항)

#### 11.1 개인정보보호 관련

> **⚠️ 중요**: 주민등록번호는 대한민국 개인정보보호법상 민감정보에 해당합니다.

- 주민등록번호는 반드시 암호화하여 저장·처리해야 합니다.
- 처리 완료 후 서버에 저장된 원본 데이터는 즉시 삭제해야 합니다.
- 동의서 출력 시 주민등록번호 뒷자리는 마스킹(*******) 처리합니다.
- 접근 로그를 기록하여 감사 추적(audit trail)을 지원해야 합니다.
- 개인정보 처리방침을 시스템 내에 명시해야 합니다.

#### 11.2 전산 시스템 연동 관련

> **⚠️ 중요**: 현재 요구사항에는 "웹 브라우저 사이트에 접속해서" 전산 등록을 진행하는 부분이 있습니다.

- 대상 전산 시스템의 URL과 로그인 정보가 필요합니다.
- 전산 시스템이 API를 제공하는 경우 → API 연동으로 자동화 가능
- 전산 시스템이 API를 제공하지 않는 경우 → 브라우저 자동화(Playwright/Puppeteer)를 고려해야 하나, **Cloudflare Workers 환경에서는 브라우저 자동화가 불가능**합니다.
- 대안: 사용자의 브라우저에서 직접 전산 시스템을 열고, 데이터를 클립보드 또는 DOM 조작으로 입력하는 **하이브리드 방식** 검토 필요

#### 11.3 Cloudflare Workers 제약사항

- 파일 시스템 접근 불가 (`fs` 모듈 사용 불가)
- 실행 시간 제한 (무료 플랜 10ms CPU, 유료 30ms~)
- 메모리 제한 128MB
- Node.js 네이티브 모듈 사용 불가

### 16. 데이터 보호 영향 평가 (DPIA) 템플릿

#### 부록 A: DPIA 문서 구조

1. **처리 활동 개요**
   - 시스템명: 고객정보 등록 및 동의서 출력 업무 자동화 시스템
   - 처리 목적: 보험계약 상담·체결·유지를 위한 고객 개인정보 처리
   - 데이터 유형: 주민등록번호(민감정보), 이름, 연락처
   - 처리 규모: 배치당 최대 1,000건, 일 최대 5,000건 예상

2. **처리의 필요성 및 비례성 평가**
   - 필요성: 보험계약 체결을 위한 법적 필수 정보(주민번호), 본인 확인
   - 비례성: 최소한의 데이터만 수집(3개 필드), 불필요 컬럼 무시
   - 대체 수단: 주민번호 대체 가능성 검토(CI값, 여권번호 등)

3. **데이터 주체 권리 및 자유에 대한 위험**
   - 신원 도용 위험 (주민번호 유출 시)
   - 프라이버시 침해 위험
   - 차별적 프로파일링 위험 (보험 심사)

4. **위험 완화 조치**
   - 기술적: AES-256 암호화, HTTPS, 마스킹, 자동 삭제
   - 조직적: 접근 통제, 처리자 교육, 비밀유지 서약
   - 계약적: DPA 체결, Cloudflare DPA 확인

5. **잔여 위험 평가**
   - 분류: 중간(Medium)
   - 사유: 암호화 불가능한 전송 구간(사용자 브라우저-서버 간), 직원 실수 가능성

6. **DPO 의견** (작성 필요)

7. **최종 승인** (컨트롤러 승인 필요)
